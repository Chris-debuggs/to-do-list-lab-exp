# Phase 2: Semantic Search & Schema Expansion Architecture

The application is transitioning into a client-side Machine Learning environment. This requires massive architectural shifts to prevent the main thread from blocking during matrix multiplications and schema upgrades.

## Proposed Architecture

### 1. IndexedDB Schema Upgrade
We must abandon the primitive string/boolean data model and implement a standard structured record layout:
- **`createdAt`**: Unix timestamp (number)
- **`priority`**: Enum (`'Low' | 'Medium' | 'High'`)
- **`embedding`**: High-dimensional vector array (`number[]` or `Float32Array`) generated by the NLP model

**Mechanism:** We will increment `DB_VERSION` to 2 in [src/db/indexedDB.ts](file:///c:/Users/cnevi/Documents/to-do-list-lab-exp/src/db/indexedDB.ts) and handle the migration logic safely within the upgrade callback.

### 2. Isolated WebAssembly Inference (`src/worker/ml.worker.ts`)
To prevent the 20MB+ model from freezing the DOM, we will run `@xenova/transformers` strictly inside a Web Worker.
- The React application will dispatch text to the worker via `postMessage()`.
- The worker will download and cache `Xenova/all-MiniLM-L6-v2`, execute the inference, and yield the dense vector representation back to the main thread.

### 3. Reducer & UI Expansion
- **Purge Completed:** A new `PURGE_COMPLETED` action will delete all checked items in batch.
- **Skeleton Loaders:** An explicit loading state will render while the initial [getTodosFromDB()](file:///c:/Users/cnevi/Documents/to-do-list-lab-exp/src/db/indexedDB.ts#35-39) promise is pending.
- **Semantic Search Engine:** A secondary search bar will filter tasks not by substring matching, but by iterating over the stored database records and computing the Cosine Similarity between the search query tensor and the stored task tensors.
